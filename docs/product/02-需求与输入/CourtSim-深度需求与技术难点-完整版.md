## 一、产品定位与核心价值观

### 1.1 CourtSim 是什么

CourtSim（朝堂风云）是一款以中国古代朝堂政治为核心场景的 **历史推演引擎**。它不是历史知识测验，也不是单纯的文字游戏，而是一个让用户在有历史约束的叙事空间中扮演角色、操纵势力、体验政治博弈的互动仿真平台。

其本质是将三个传统上分离的领域融合到同一个系统中：

- **历史学的严谨性**：人物、官职、年号、地理、典故均有史料依据
- **戏剧学的张力感**：每一场廷辩都有明确的冲突弧线，有开场、升级、皇权介入、结局分支
- **游戏设计的代入感**：用户通过主视角角色在权力网络中做出影响历史走向的决策

### 1.2 三种使用模式

**严格模式（Strict Mode）**：仅允许有史可查的人物、事件和言论逻辑。LLM 的任何输出都要经过 RAG 史料约束，幻觉应在提示层被拦截。适合历史研究者、教育场景。

**演绎模式（Deductive Mode）**：在史实框架内允许合理推演。人物可说未见于史册的话，但其态度、立场、行为方式必须符合历史记载的人物侧写。这是默认模式，平衡娱乐性与准确性。

**架空模式（Counterfactual Mode）**：允许改变历史结果，但要求用户承认这是「假如」推演。例如「假如于谦在夺门之变前获得密报，他会如何应对？」。模型可以自由发挥，但仍须维持明代中期的政治文化生态。

三种模式在 `scenario.simulationMode` 字段中通过枚举区分，并影响 RAG 检索的强制程度和提示中的约束力度。

## 二、已识别的困难点（深度分析）

### 2.1 历史真实性与 LLM 幻觉

**成因分析**

LLM 在历史细节上的幻觉有几种典型模式：

- **朝代错位**：将明代制度（如内阁票拟）套用到唐代，或把宋代官职名称用于清代语境
- **人物混淆**：同名/近名人物相互污染（如明代两位王守仁、多位杨廷和、杨一清）
- **年号错乱**：模型对年号与皇帝的对应关系经常出错，尤其在一帝多年号（如明英宗天顺/正统）情况下
- **地理失真**：行政区划随朝代变化剧烈，明代「南直隶」在模型中常被与清代「江苏」混用
- **职能幻觉**：模型知道「都察院」的名称，却可能不清楚其左都御史与右都御史在廷议权限上的差异

**风险评级**

在严格模式下，单次推演中出现至少一处可被专业人士识别的史实错误的概率估计超过 60%（基于对同类历史文本生成任务的观察）。这对以教育为场景的部署是不可接受的。

**缓解策略**

第一层：**结构化人物档案**。每个进入推演的历史人物都应有一份经过人工校对的元数据档案，包含：任职年表（带年号范围）、官职变迁、派系归属、主要政治主张、语言风格特征。模型在生成角色对话时以此档案为硬约束。

第二层：**RAG 史料管道**。在每次角色发言前，系统检索该角色在相关议题上的历史记录（奏疏、实录、文集片段），注入到角色扮演提示的上下文中。这不是为了让模型「照抄」史料，而是为模型提供「这个人在这类问题上的实际态度」的证据锚点。

第三层：**幻觉后验检测**。在生成完成后，用一个轻量级的结构化校验提示检查输出中的官职、年号、人名是否在该 scenario 的时间窗口内合法。这层检测可以是异步的，不阻塞主流程，但检测结果应记录到 `simulationRun.qualityFlags` 字段（⚠️ 建议变更：当前 schema 中可能无此字段，建议添加）。

第四层：**模式差异化约束**。严格模式下，提示中明确禁止模型引入未在档案或史料中出现的细节；演绎模式下，约束放宽为「人物行为须符合其历史侧写」；架空模式下，仅保留朝代制度框架的约束。

### 2.2 长上下文成本

**成因分析**

CourtSim 的核心体验是多角色多轮廷辩。一场典型的「大礼议」推演可能包含：

- 8–12 个参与角色，每轮每角色平均发言 200–400 token
- 10–20 轮推演
- 每轮携带的历史背景上下文约 1000–2000 token
- RAG 检索结果注入约 500–1500 token/轮

粗略估算：20 轮 × 10 角色 × 300 token/发言 = 60,000 token 纯对话内容，加上重复注入的背景上下文，总上下文可能轻松超过 100,000 token。在当前主流 LLM 定价体系下，单次完整推演的 API 成本可能达到数美元，对 C 端产品是致命的。

**语义损失风险**

截断策略（直接丢弃早期轮次）在廷辩推演中尤为危险，因为：

- 早期轮次建立的「政治债务」（某官员曾当众表态支持某立场）在后续轮次中可能被其他角色引用
- 皇帝的中期表态往往是后续所有角色行为的约束边界，若被截断则模型无法正确处理

**缓解策略**

**轮次摘要策略**（当前已部分实现）需要精细化：

普通摘要记录「谁说了什么」，但 CourtSim 需要的是「政治状态摘要」，具体包含：

- 各势力当前的公开立场（已表态/未表态/模糊表态）
- 皇帝已释放的倾向信号（若有）
- 已发生的关键事件（有人被当廷斥责、有人告退、有人上疏）
- 尚未解决的核心议题

这种结构化摘要比自由文本摘要的 token 效率高 3–5 倍，且在回代注入时对模型的指导更精确。

建议在 `RoundSummary` 模型中增加 `politicalState` JSON 字段，结构如下：

```json
{
  "factionStances": {
    "礼仪派": "反对",
    "实用派": "支持",
    "中立": ["某官员A", "某官员B"]
  },
  "emperorSignals": ["表示将召集廷议", "对张璁明显亲近"],
  "keyEvents": ["杨廷和当堂拂袖而去"],
  "openQuestions": ["世宗生父究竟如何尊称"]
}
```

**分层上下文架构**：

- **永久层**（不压缩）：scenario 元数据、人物档案、朝代制度框架，约 3000–5000 token
- **滑动窗口层**（保留最近 N 轮完整对话，N 可配置，默认 3）：约 10,000–20,000 token
- **摘要层**（滑动窗口之外的历史轮次的政治状态摘要）：每轮约 200–300 token，10 轮约 2000–3000 token
- **RAG 注入层**（每轮按需检索，用完即丢）：约 500–1500 token

这一架构可将 20 轮推演的有效上下文稳定在 30,000 token 以内。

### 2.3 RAG 与戏剧冲突

**核心矛盾**

史料的特点是客观、克制、事后叙述；廷辩戏剧的特点是当下性、情绪性、对抗性。直接将《明实录》片段注入角色提示，往往导致模型生成「报告文体」而非「戏剧对白」。

典型失败模式：

> 「臣以为，据《礼记》所载，……此议可从。」（正确但无戏剧张力）

期望模式：

> 「杨廷和冷笑，缓缓起身：『诸公说得轻巧。若依张璁之议，则先帝何处？』」

**缓解策略**

RAG 检索结果不应直接注入到角色的「即将说话」提示中，而应作为角色的「内在知识储备」注入到角色侧写提示的独立区块中，并明确指令：「以下是该角色的历史立场证据，用于约束你的发言方向，但你的表达方式应符合戏剧情境，而非复述史料。」

此外，为每个场景预置「戏剧冲突催化剂」——一组能引发立场升级的触发词和情境描述。当推演进入平淡区间时，系统可以注入这些催化剂。

### 2.4 论辩结构可控性

**症状**

即使在精心设计的提示下，多角色廷辩仍有三个常见退化模式：

- **论据趋同**：多个不同立场的角色最终使用相似的引经据典方式，论辩缺乏差异感
- **套路化礼让**：角色倾向于在批评前加大量铺垫（「臣愚见……」「圣恩浩荡……」），稀释了冲突锋芒
- **皇权虚化**：皇帝角色要么沉默（模型不知道何时该皇帝出手），要么越权（模型把皇帝写成替臣子作结论）

**皇权介入机制设计**

皇权在真实廷议中的功能不是「做出最优决策」，而是「管理信息、控制节奏、保持所有势力对自己的依附」。这一政治逻辑必须通过提示设计传递给模型。

具体方案：为皇帝角色设置独立的「皇权行为菜单」，包含：

- **垂询**：就某一议题向特定大臣单独问询，打破联合
- **留中**：搁置奏疏，保持压力
- **廷杖**：实施惩罚，震慑场域
- **特旨下发**：绕过廷议直接决定，打破僵局
- **暗示倾向**：通过赏赐、接见、语气变化释放信号

这些行为不应由模型自主决定，而应由推演引擎在特定条件下（如连续 N 轮无实质推进）触发「皇权介入点」，并向用户或皇帝 Agent 提示可选行为。

### 2.5 SQLite 向量规模

**技术背景**

当前实现使用 Prisma Bytes 存储向量，全表余弦扫描。在史料库规模较小（数千条）时可接受，但随着内容增长，性能会线性退化。

**规模估算**

一个覆盖明清两代主要政治事件的史料库，保守估计需要：

- 人物档案：500–1000 条，每条约 500 token
- 史料片段（奏疏、实录、文集）：10,000–50,000 条，每条约 200–800 token
- 场景描述与地理背景：1000–3000 条

以 50,000 条向量（维度 1536）估算，全表扫描单次检索约需 150–300ms（取决于硬件），在 SSE 流式推演中处于可接受边界，但有明显抖动风险。

**迁移决策触发点**

建议以「史料条数超过 20,000 条」或「检索 P95 延迟超过 200ms」作为迁移触发条件。

详见 [技术决策点 3.1](#31-向量存储)。

### 2.6 多实例 SSE

**问题描述**

当前架构使用进程内事件总线推送 SSE。这意味着：当部署扩展到多个 Node.js 进程（如 Vercel 多 Region、PM2 集群、K8s 多 Pod）时，一个进程产生的推演事件无法被连接到其他进程的客户端接收。

**风险量化**

对于个人/小团队使用场景，单进程部署是可以接受的。但一旦进入多用户并发场景，该架构会导致随机出现的「推演消失」Bug，表现为客户端长时间 pending 但无事件推送。

**缓解策略**

详见 [技术决策点 3.3](#33-sse-多实例方案)。

### 2.7 关系网络数据稀疏

**问题根源**

`relationships` JSON 字段的数据来源有两个渠道：

1. 用户在向导中手动建立
2. 从推演生成流程中自动提取

渠道 1 依赖用户主动性，大多数用户不会在正式推演前手动配置全部关系；渠道 2 需要明确的提取机制，当前尚未完整实现。

**自动提取方案**

在每轮推演完成后，对 LLM 生成内容进行后处理，提取以下关系事件：

- 明确的支持/反对表态（「臣支持张璁之议」→ 增加/增强正向关系边）
- 人身攻击或斥责（→ 增加/增强负向关系边）
- 结盟暗示（「与某公共同上疏」→ 确认盟友关系）

这些提取结果可以用轻量级结构化提示完成，不需要完整 LLM 推理。

**初始化方案**

为内置的主要场景（如大礼议、夺门之变）预置关系网络初始状态，基于历史记载。这样即使用户不配置，关系图也不会空心。

### 2.8 章回体风格漂移

**症状分类**

- **现代网文化**：出现「眼神变得凌厉」「内心一阵波动」「心中冷哼」等网络小说惯用句式
- **过度文言化**：模型为显示「古风」而强行使用生僻文言，导致语义晦涩甚至错误
- **结构趋同**：每章开头都是环境描写，中间都是对话，结尾都是悬念钩子，缺乏变化

**标准化策略**

为章回体报告建立一份「风格宪章」，包含：

- **允许使用的文学手法**：白描、插叙、场景切换
- **禁用的表达模式**：心理活动过度描写、现代情绪词汇（「崩溃」「破防」「内耗」）、「不明觉厉」式悬念
- **参考文本**：《明史》纪传体叙事段落、《三言两拍》白话叙事片段（按场景类型选择）
- **章回套语规范**：开章套语（「话说……」「却说……」）和收章套语（「欲知后事如何，且听下回分解」）的使用频率与变体

这份风格宪章以系统提示形式注入，并在每次生成章回时附带 2–3 个精选的同类型示例（few-shot 示例，而非风格说明）。

### 2.9 主视角与多 Agent 一致性

**叙事悖论**

CourtSim 的主视角机制声称用户只通过一个角色的视角体验推演，但后端实际上是多 Agent 并行推理（每个历史人物都是独立的 LLM 调用）。这产生了一个叙事一致性问题：

其他 Agent 拥有「上帝视角」（能看到所有角色的发言），但主视角角色在历史上应该有信息限制（并非所有廷议内容都公开，私下密谋不可能被对手获知）。

**解决方向**

引入「信息层级」机制：

- **公开信息**：所有廷议记录、明发谕旨，全体 Agent 均可获取
- **半公开信息**：小规模传播的内部奏疏、私下串联，只有特定派系 Agent 可获取
- **私密信息**：个人密谋、皇帝密旨，仅当事人 Agent 可获取

主视角 Agent 只接收主角色按其政治地位和信息渠道应能获取的信息。其他 Agent 的「私密层」不注入主视角的上下文。

这一机制的实现意味着每轮推演需要为不同 Agent 构建不同的上下文包，成本略高，但对沉浸感的提升是根本性的。

### 2.10 国际化与敏感叙事

**教育场景合规风险**

若 CourtSim 面向 K-12 或高校教育场景，以下内容需要特别处理：

- **宫廷暴力**：廷杖、抄家、株连的描写需要可配置的「过滤级别」
- **政治污名化**：对历史人物的道德评价应区分「历史评价（史书记载的评价）」和「产品生成的新评价」
- **敏感事件**：涉及明清之际族群冲突、近代史衔接部分需要特别的编辑政策

**国际化**

英语用户群体对「廷议」「票拟」「奉旨」等术语的理解需要依赖术语表和上下文解释。建议国际版在首次出现专名时自动添加悬停解释，而非全文翻译（后者会破坏沉浸感）。

## 三、技术决策点（方案对比）

### 3.1 向量存储


| 方案                                       | 优势                                                        | 劣势                                          | 适用阶段                        |
| ---------------------------------------- | --------------------------------------------------------- | ------------------------------------------- | --------------------------- |
| **Prisma Bytes + 全表扫描**（当前）              | 零运维、与现有 ORM 无缝集成、本地开发友好                                   | 性能随数据量线性退化；无近似算法支持                          | 史料 < 10,000 条、单用户/小团队       |
| **sqlite-vss**（推荐过渡方案）                   | 仍是 SQLite 文件、无额外服务依赖、支持 HNSW 近似搜索、Prisma 可通过 raw query 调用 | 需要 native addon，Vercel 等无服务器环境需特殊处理；社区活跃度较低 | 史料 10,000–100,000 条、单机或容器部署 |
| **pgvector + PostgreSQL**（推荐生产方案）        | 工业级稳定性、Prisma 原生支持、与主数据库同服务可简化运维                          | 需要迁移到 PostgreSQL（涉及 schema 变更）；本地开发复杂度增加    | 多用户、史料 > 50,000 条、生产环境      |
| **专用向量服务**（Pinecone / Weaviate / Qdrant） | 最高检索性能；支持元数据过滤（按朝代、人物、事件类型）；托管版无运维负担                      | 额外服务依赖和费用；数据不在同一存储中增加一致性维护成本                | 大规模多租户、史料 > 500,000 条       |


**决策建议**：当前阶段维持现有方案；当史料条数超过 15,000 条时，优先考虑迁移到 pgvector（同时将主数据库从 SQLite 升级到 PostgreSQL），一步到位，避免二次迁移。专用向量服务在现阶段是过度设计。

**元数据过滤的重要性**：无论选择哪个方案，向量检索的元数据过滤能力至关重要。CourtSim 的检索通常需要按朝代+时间范围+角色ID+议题类型组合过滤，纯向量相似度搜索无法满足这一需求。建议始终在向量检索后加一层元数据过滤。

### 3.2 摘要策略


| 策略                     | 质量                   | 延迟                      | 成本  | 备注                      |
| ---------------------- | -------------------- | ----------------------- | --- | ----------------------- |
| **每轮 LLM 摘要**（当前）      | 高（语义连贯）              | 中（+1 LLM 调用/轮）          | 中   | 需要专门的「政治状态摘要」提示         |
| **滑动窗口截断**             | 低（早期政治债务丢失）          | 最低                      | 最低  | 适合快速 Demo，不适合正式推演       |
| **关键句抽取**（TF-IDF/BM25） | 中（保留信息量大的句子，但失去政治逻辑） | 低                       | 极低  | 可作为 LLM 摘要的前处理，降低摘要输入长度 |
| **结构化状态追踪**（推荐）        | 最高（明确记录政治状态变量）       | 低（状态更新可规则化，无需完整 LLM 推理） | 低   | 需要预定义状态变量 schema，工程量较大  |


**推荐组合**：结构化状态追踪（记录 `factionStances`、`emperorSignals`、`keyEvents`）+ 最近 3 轮完整对话保留 + 再早的轮次仅保留政治状态摘要。LLM 完整摘要仅在章回报告生成时使用，不在推演中间轮次使用。

### 3.3 SSE 多实例方案


| 方案                           | 优势              | 劣势                             | 适用场景                |
| ---------------------------- | --------------- | ------------------------------ | ------------------- |
| **进程内总线**（当前）                | 零依赖、最简实现        | 不支持横向扩展；进程重启丢失状态               | 单进程部署、开发阶段          |
| **Redis Pub/Sub**            | 轻量级、延迟极低、广泛支持   | 需要 Redis 实例；消息无持久化（进程重连可能丢失事件） | 生产多实例、延迟敏感          |
| **数据库轮询**（Prisma + interval） | 无额外服务依赖；事件天然持久化 | 延迟较高（取决于轮询间隔，通常 500ms–2s）      | 对延迟不敏感、希望最小化基础设施的场景 |
| **服务端 Webhook + 客户端重连**      | 简单、容错性强         | 开发体验较差；每次重连需要回放历史事件            | 移动端、不稳定网络环境         |


**决策建议**：对于 Vercel 部署（无状态函数），进程内总线从根本上不可行，应改用**数据库轮询方案**（将推演事件写入 `SimulationEvent` 表，客户端通过 SSE endpoint 定期轮询新行），这与 Prisma SQLite 的现有架构完美契合，不引入新依赖。若迁移到有状态容器部署，再升级到 Redis Pub/Sub。

### 3.4 对话结构约束


| 方案                                     | 可靠性           | 复杂度                      | 备注                            |
| -------------------------------------- | ------------- | ------------------------ | ----------------------------- |
| **Prompt + dialogueMeta JSON**（当前）     | 中（模型偶尔忽略结构要求） | 低                        | 快速迭代友好                        |
| **结构化输出 schema**（JSON Mode / Tool Use） | 高（格式由模型保证）    | 中                        | 需要为每种对话类型设计 schema；不同模型支持程度不同 |
| **二次裁判模型**                             | 最高（独立验证）      | 高（+1 LLM 调用；延迟增加 30–50%） | 适合严格模式；成本较高                   |
| **规则后处理**                              | 中（只能检测已知违规模式） | 低                        | 可作为结构化输出的补充                   |


**推荐**：优先推进结构化输出 schema，将 `dialogueMeta` 从 prompt 内的自然语言要求升级为 Tool Use / JSON Schema 强约束。在严格模式下可叠加规则后处理，暂不使用二次裁判（成本过高）。

**决策建议**：现有架构（独立页 + 向导内嵌）在功能深度上占优，建议维持，但优化首访引导——增加一个「欢迎卡片」（首次使用时出现），以「选择一个历史场景」为切入点，减少空白状态焦虑。

## 四、历史推演需求案例库（完整版）

### 4.1 政变类

#### 明 · 夺门之变（1457）

**历史背景**

景泰八年（1457）正月，景帝朱祁钰病重，储位空悬。武清侯石亨、太监曹吉祥、副都御史徐有贞等人策划拥立被软禁于南宫的前英宗朱祁镇复辟。正月十七日凌晨，石亨等人破南宫门，奉英宗直入奉天门，英宗重登宝座，史称「夺门之变」（英宗自称「夺门」，以彰显正当性；于谦系则称之为「南宫复辟」）。

**登场势力**

- **复辟派**：石亨（手握京营兵权）、曹吉祥（司礼监太监，掌印）、徐有贞（谋划核心，事后任内阁）、张軏（配合控制禁军）
- **景帝—于谦系**：于谦（兵部尚书，土木堡后北京保卫战的主导者）、王文（大学士，景泰朝核心辅臣）、景帝朱祁钰（此时已病重，基本丧失行动能力）
- **观望派**：大多数勋贵和朝官，待势力明朗后随风转向

**核心议题与胜负手**

核心议题有三：储位合法性（景帝子已死，储位悬而未决）、兵权归属（石亨控制关键部队）、政治信息不对称（于谦系不知复辟计划的具体时间节点）。

胜负手在于：夺门派选择在景帝最虚弱、于谦系无法紧急动员的凌晨窗口行动，将时间差转化为政治既成事实。于谦获报后，以「社稷为重，今上（英宗）已正位」为由，拒绝以武力对抗——此举既显示其政治道德，也注定了他的悲剧命运。

**史料建议**

- 《明英宗实录》天顺元年正月条
- 《明史》卷一七〇《于谦传》
- 《菽园杂记》对事件现场的记述
- 《国榷》谈迁对夺门正当性的争议记录

**CourtSim 推演场景设计**

可设计两个推演节点：

节点 A（事前）：景泰七年末，景帝病情趋重，廷议如何处置储位——此时于谦系、复辟派、中立派三方都在试探。

节点 B（事中/事后）：正月十七日清晨，英宗已入奉天门，消息传开，各方大臣须当场决定是否叩贺。这一场景的戏剧张力在于：没有人知道形势是否已经不可逆转，每个人都在等别人先表态。

#### 唐 · 玄武门之变（626）

**历史背景**

武德九年（626）六月初四，秦王李世民在玄武门伏击太子李建成和齐王李元吉，两人被杀。高祖李渊随后被迫立李世民为太子，旋即禅位。这是中国历史上被研究最多的宫廷政变之一，其历史叙述本身也充满了有意为之的再建构。

**登场势力**

- **秦王府集团**：李世民、长孙无忌、房玄龄、杜如晦、侯君集、尉迟恭（武将执行层）
- **东宫集团**：李建成、魏征（后投李世民）、王珪、薛万彻
- **齐王府**：李元吉（与东宫联合，是李世民最直接的军事威胁）
- **高祖集团**：李渊及少数仍忠于他的朝臣，在政变发生时完全失去行动能力

**CourtSim 特殊处理**

玄武门之变的政治博弈主要发生在事前（武德七年后的三方角力），而非事发当日。推演可聚焦于：

- 李世民阵营的内部决策：「继续被动等待」vs「先发制人」
- 朝臣立场的撕裂：谁是关键的中间派，其一票倒向哪方
- 信息战：李建成和李元吉对秦王府异动的侦知与误判

---

#### 清 · 辛酉政变（1861）

**历史背景**

咸丰帝驾崩于热河，遗命载垣、端华、肃顺等八大臣辅政。慈禧太后（懿贵妃）与恭亲王奕訢联手，利用顾命大臣回京护送梓宫的窗口期，发动政变，逮捕并处决肃顺等人，建立两宫垂帘听政制度。

**CourtSim 特殊价值**

辛酉政变是研究清代宫廷权力结构的绝佳案例，因为它涉及：

- 满汉权力关系（肃顺系的满洲保守派 vs 奕訢的汉化洋务派）
- 后宫与议政的制度边界（垂帘听政是对清祖制的突破）
- 地缘因素（热河—北京的空间距离如何影响政变时机）

#### 宋 · 陈桥兵变（960）

**历史背景**

后周显德七年（960）正月，征北汉途中，赵匡胤部队发生「兵变」，将士以黄袍加身，赵匡胤被迫称帝，建立宋朝。这一「被迫」的叙事框架是赵匡胤精心设计的政治话语，现代史学者普遍认为这是一次预谋性政变。

**推演特色**

陈桥兵变的精髓在于「信息管控」和「合法性建构」，而非军事对抗。推演可以重点模拟：

- 行军途中的谣言制造与传播（「北方有两个太阳」之说的刻意散布）
- 赵匡胤本人的「被动配合」表演（如何让「被迫」看起来真实）
- 如何在最短时间内令后周朝廷接受既成事实

### 4.2 改革类

#### 北宋 · 熙宁变法（1069–1076）

**历史背景**

神宗熙宁二年，王安石以参知政事身份主持变法，推行青苗法、募役法、市易法、保甲法、将兵法等一系列改革，意图增加国家财政收入、减轻土地兼并、加强边防。改革触动了大地主、大官僚和旧有利益集团，遭到以司马光为代表的元祐党人反对，史称「新旧党争」。

**登场势力**

- **变法派**：王安石（变法核心）、吕惠卿（王安石的得力助手，后与王安石反目）、曾布（早期支持，后倒戈）、神宗皇帝（关键支持者，但在压力下摇摆）
- **保守派**：司马光（核心反对者）、韩琦（元老）、欧阳修（对青苗法尤为激烈批评）、苏轼兄弟（对变法持复杂态度）
- **台谏**：御史台和谏院的官员在变法中扮演了核心角色，是两派争夺的焦点

**核心议题与冲突结构**

青苗法是最具代表性的冲突场域。支持方认为：以低利贷款替代民间高利贷，既增加国库收入又保护农民。反对方认为：地方官强行摊派，实质上增加了农民负担；而且「国家做生意」破坏了传统政治伦理。

CourtSim 推演的精华在于：让用户扮演一个对变法利弊有疑问的中层官员（如提点刑狱），在两派压力下做出站队或周旋的决策。

**史料建议**

- 《宋史》卷三二七《王安石传》
- 《续资治通鉴长编》熙宁年间各卷
- 《临川先生文集》王安石论青苗法奏疏
- 《传家集》司马光批评青苗法的奏议

---

#### 北宋 · 庆历新政（1043–1044）

**历史背景**

仁宗庆历三年，范仲淹以参知政事身份提出「十事疏」，包括明黜陟、抑侥幸、精贡举、择官长等十项改革建议。新政仅推行一年余即遭打压，范仲淹被贬。庆历新政是熙宁变法的历史前奏，但改革力度和制度准备均不及后者。

**推演特色**

庆历新政的失败有一个值得深究的结构性原因：仁宗皇帝的「慈善性软弱」——他对范仲淹有真实好感，但在保守派的持续压力和皇后集团的干预下，始终无法给予决定性的政治保护。推演可以模拟这一结构性困境：改革者即使拥有皇帝的初始支持，如何在宫廷政治中保持这种支持？

#### 唐 · 永贞革新（805）

**历史背景**

唐顺宗永贞元年，王叔文、王伾、柳宗元、刘禹锡等人主导的改革，试图削减藩镇势力、整顿宦官专权、打击地方豪强。改革仅持续约半年，因宦官联合藩镇反扑而失败，顺宗被迫禅位。

**推演特色**

永贞革新是「彻底的改革理想与脆弱的权力基础之间的悲剧」。顺宗病重（疑似中风，无法正常理政）这一生理因素本身就使改革的政治基础极不稳固，为推演提供了罕见的「主君健康状况影响政治走向」这一变量。

#### 清 · 戊戌变法（1898）

**历史背景**

光绪二十四年，光绪帝在康有为、梁启超等维新派推动下颁布「明定国是」诏书，开展政治、教育、军事变法，史称「百日维新」。慈禧太后发动「戊戌政变」，囚禁光绪，处决谭嗣同等六人（「戊戌六君子」），废除新政。

**推演特色**

戊戌变法的推演价值在于权力中枢的双轨结构：名义上的最高权力者（光绪帝）和实质权力掌握者（慈禧）之间的张力，使所有政治行为者都必须在「效忠谁」这一根本问题上做出抉择。

### 4.3 朝堂政治类

#### 明 · 大礼议（1521–1536）

**历史背景**

正德十六年，武宗驾崩无嗣，杨廷和等大臣迎立藩王朱厚熜即位（嘉靖帝）。随即引发长达十余年的「大礼议」：嘉靖坚持尊其生父兴献王为皇考，而以杨廷和、毛澄为首的大臣坚持嘉靖应以孝宗为皇考、生父只能称「皇叔父」。最终嘉靖以张璁等新进官员为援，强行推行自己的主张，大批反对派官员被廷杖、贬黜。

**推演价值**

大礼议表面是礼制之争，实质是政治权力的重新划分：谁能定义「礼制」，谁就掌握了最高政治话语权。嘉靖通过坚持礼议，成功打倒了正德遗留的内阁势力，重建了皇权直接统治的格局。

这是 CourtSim 中推演价值最高的场景之一，因为它几乎包含了所有朝廷政治博弈的要素：礼制论证、人身风险（被廷杖）、派系形成与崩解、皇帝作为最终仲裁者的角色、时间拉锯战。

---

#### 明 · 国本之争（1586–1620）

**历史背景**

万历帝长子常洛（郑贵妃所生为福王常洵）与神宗宠妃郑贵妃推动的「国本」（皇储）问题，持续三十余年，耗尽万历朝政治能量。神宗以各种理由拖延立储，大臣们前赴后继上疏力争，被罚、被贬者不计其数。

**推演特色**

国本之争的核心冲突是「祖宗家法」（以长立储）与「皇帝个人意志」之间的正面交锋。这一场景特别适合展示明代朝廷政治文化中「廷议作为道德法庭」的功能：大臣们明知反对皇帝的成本极高，却前赴后继，因为「坚持」本身就是政治资本积累。

#### 唐 · 牛李党争（9世纪）

**历史背景**

唐代中后期，以牛僧孺为代表的「牛党」与以李德裕为代表的「李党」长期对立，两党的核心分歧涉及：科举出身 vs 门荫出身的仕途通道之争、对藩镇的态度（妥协 vs 强硬）、以及纯粹的权力争夺。这场党争延续约 40 年，严重消耗了唐朝的政治能量。

**推演特色**

牛李党争的独特价值在于它展示了「意识形态分歧与权力分歧的缠绕」：两党表面上争论政策，实质上是两个社会群体（寒门科举新贵 vs 士族门阀后裔）的政治存亡之战。

---

#### 北宋 · 澶渊之盟前后主战主和（1004）

**历史背景**

辽军大举南侵，兵锋直指澶州（开封北方门户）。真宗朝廷中，寇准主张皇帝亲征以鼓舞士气，王钦若等人主张迁都或割地求和。最终真宗在寇准力争下御驾亲征，与辽签订「澶渊之盟」，以岁币换取和平。

**推演特色**

这一场景是「短期代价 vs 长期利益」政治辩论的经典案例：主战派（寇准）以道德话语压制主和派，主和派则以实际风险分析应对。推演可让用户体验「如何在道德高地被占领后仍为合理政策辩护」这一困境。

---

### 4.4 军事—财政联动

#### 明 · 土木堡后京军重建与边防（1449–）

**历史背景**

土木堡之变，明军主力和英宗本人均遭捕获。于谦主持的北京保卫战胜利后，军队重建、边防调整和财政重组成为首要任务。这一场景涉及兵部（军事）、户部（财政）、内阁（决策）之间的协调与摩擦。

**推演特色**

这一场景的推演价值在于「危机后的制度重建」——在资源有限的情况下，各部门如何分配权力和财政支持。此外，这一时期还有复杂的人事政治：谁因土木堡而受益（于谦系），谁因此而怀恨（英宗复辟后清算于谦的根源）。

#### 清 · 甲午主战主和与海军经费（1894）

**历史背景**

光绪朝甲午战争前后，围绕是否与日本开战、如何运用北洋海军，朝廷中主战派与主和派形成对立。而海军经费被挪用（传统说法认为部分资金流入颐和园修缮）是北洋海军备战不足的重要原因之一。

**推演特色**

这一场景融合了军事战略辩论、财政优先级争议和后宫政治影响三个维度，同时具有强烈的历史反思价值——推演结果无论主战还是主和，都面临真实历史中的沉重结局。

### 4.5 主视角 RPG 向（任务树设计）

#### 场景一：低阶御史在大礼议中站队与保命

**角色设定**

主角：某六品监察御史，资历尚浅，无强大派系背景。初入京城不久，对礼议之争有自己的判断，但同时清楚错误站队的风险。

**任务树（三叉结构）**

```
开局任务：获取情报
├── 支线A：拜访杨廷和派系大臣 → 被拉拢入反嘉靖阵营
│   ├── 选择服从 → 上疏批评张璁 → 被廷杖（受伤但获声望）→ 事后贬谪风险
│   └── 选择观望 → 被派系踢出 → 独立生存困难加剧
├── 支线B：结交张璁新进 → 获得皇帝注意机会
│   ├── 选择主动上疏支持嘉靖 → 快速晋升 → 杨廷和党清算风险（若日后反转）
│   └── 选择等待时机 → 在嘉靖最终胜势明朗时出手 → 晚一步但更安全
└── 支线C：中立回避 → 专注弹劾具体腐败案 → 维持生存但错过历史机遇
```

**关键决策点**

第三轮：有大臣在廷杖后死亡，此时是否上疏抗议？（高风险/高声望 vs 沉默保命）

**结局分支**

- 「明智存活」：中立生存，在嘉靖胜出后以「未曾反对」为由获得有限晋升
- 「殉道留名」：廷杖后不屈，贬谪多年，事后平反
- 「新贵捷径」：早期押注张璁成功，快速升迁，但政治生命与张璁绑定

---

#### 场景二：藩王近臣在靖难前夕的信息博弈

**角色设定**

主角：燕王府（朱棣）的一名幕僚，洪武三十一年建文帝即位后，诸藩削藩阴云笼罩。主角必须在燕王「是否举兵」这一生死决断前，完成信息收集、立场研判和自身处置。

**任务树**

```
阶段一：情报收集（靖难前二年）
├── 北平本地网络：掌握朝廷削藩动态
├── 南京内线：与建文朝臣中的同情者建立联系
└── 燕王府内部：理解朱棣的真实意图（他是真的犹豫还是在等时机？）

阶段二：建议时机（靖难前半年）
├── 建议举兵 → 若成功：大功封赏；若失败：诛族
├── 建议继续隐忍 → 被朱棣怀疑动摇，可能被排除核心圈
└── 沉默等待 → 处于信息被动位置，错失决断机遇

阶段三：靖难爆发后的选择
├── 留守北平（高风险）
├── 随军出征（高回报）
└── 秘密南下（背叛选择，后果严峻）
```

---

#### 场景三：户部司官在变法催科与赈济之间的夹缝

**角色设定**

主角：熙宁变法期间某省的户部司官，同时承受来自「变法派」的催收青苗贷款的压力和本省遭遇旱灾需要赈济的实际需求。

**核心冲突**

- 上级要求：按时催收青苗钱，完成指标
- 地方现实：旱灾导致农民无力偿还，强制催收将激起民变
- 制度约束：申请赈灾需要走正式程序，时间来不及；擅自缓征则违反新法

**任务树**

```
选择一：如实上报 → 等待朝廷回应 → 灾情恶化 / 可能被追责「失职」
选择二：虚报完成 → 借债垫付 → 个人财务风险 / 短期过关
选择三：以「实地调查」名义拖延 → 争取时间灰色操作 → 被御史弹劾风险
选择四：联署同僚集体上疏 → 触怒变法派 → 集体被贬 / 但问题被迫正视
```

**这一场景的设计价值在于**：展示历史改革的「末端执行困境」——任何宏观改革政策，在落地时都会遭遇与顶层设计完全不同的现实约束。没有纯粹正确的选择。

## 五、名场面剧本纲要（10+ 条）

### 剧本 1：「南宫破门」

**场景**：景泰八年正月十七日凌晨，南宫（紫禁城东南一处被围堵的宫殿群），被软禁七年的英宗朱祁镇正在室内枯坐。

**人物表**

- 英宗朱祁镇（主视角，若玩家选择此角色）
- 石亨（武将，进入南宫的行动主导者）
- 徐有贞（文臣谋主，此刻在外协调）
- 英宗钱皇后（陪伴英宗，此刻在室内）
- 看守太监（信息来源）

**开场白**

宫门外传来异响。钱皇后从睡梦中惊醒，侧耳倾听。英宗起身，手按床柱，表情在惊愕与期待之间。看守太监跌跌撞撞推门而入：「陛下——不，上皇，外面……石侯爷来了。」

**冲突升级**

- 英宗是否相信这是真实机会（此前已有多次虚惊）
- 如何在短时间内完成政治重新定位：从「囚徒」切换为「皇帝」的言行举止
- 进入奉天门后，第一批前来叩见的朝臣名单，意味着什么

**皇权介入点**

此场景中「皇权」是被恢复的对象。实际的制度性介入发生在英宗重登宝座、以皇帝身份颁布第一道诏书的瞬间——这一刻系统应自动触发「新皇权宣示」事件，所有角色的政治计算须重新初始化。

**可选结局分支**

- 英宗当即表示将重用原班旧臣（试图修复与于谦的关系）→ 历史未走此路，但架空模式可以探索
- 英宗被石亨等人引导，迅速下令逮捕于谦、王文 → 历史路线
- 英宗表现软弱，被石亨等人实质架空 → 架空路线

### 剧本 2：「杨廷和的最后一票」

**场景**：嘉靖初年，大礼议最激烈的廷议之一。杨廷和已决定去职，但在离朝前发表最后一次公开表态。

**人物表**

- 杨廷和（内阁首辅，礼议反对派核心，主视角可选）
- 张璁（新进官员，嘉靖礼议的理论支柱，激进改革者）
- 毛澄（礼部尚书，杨廷和的盟友，正在动摇）
- 嘉靖帝（以诏书或私下召见形式出现，不直接上场）
- 某六品御史（见证者，可作为主视角）

**冲突结构**

杨廷和的辞职本身是一种政治表态。他的去留牵连着整个旧内阁集团的存亡。他是否「体面离场」，还是「以辞职相威胁再一次对抗」，决定了他能为盟友争取多大的政治空间。

**戏剧高潮**

张璁在廷议上当众宣读其论证生父入庙的礼制新说，逻辑严密，挑战了杨廷和集团的核心论据。杨廷和必须当场回应：

选项 A：正面辩驳（放弃离朝计划，重新投入论战）——说明他未曾真正甘心  
选项 B：沉默不应（以退为进，让张璁在无对手的情况下显得唐突）  
选项 C：点名攻击张璁出身资历（人身攻击，降低身份但有效）

---

### 剧本 3：「寇准与真宗：御驾亲征前的密室」

**场景**：澶渊之役，真宗在御驾亲征的前夜，私下召见寇准。

**戏剧核心**

真宗实际上并不想亲征。他的恐惧是真实的，他的迟疑是真实的。寇准必须在这个密室场景里，用什么语言和逻辑，让一个本质上懦弱的皇帝做出历史性的决断。

这是典型的「顾问说服君主」场景，其戏剧张力来自：真宗的每一个借口，都有其合理性；寇准的每一次回应，都在走更细的绳索。

**皇权介入方式**

此场景中皇帝是被说服的对象，「皇权」以「最终决断」的形式在密室结束时出现。系统设计建议：在寇准完成三次主要论据阐述后，触发「皇帝决断时刻」——根据推演中论据的质量分和皇帝 Agent 的状态评分，决定真宗是否同意亲征。

### 剧本 4：「熙宁三年：青苗法的第一次现实」

**场景**：青苗法颁行后的第一个春天，某地方官召集当地保甲长、里正，现场宣读青苗法条令。

**人物表**

- 地方官（主视角）
- 保甲长（代表地方精英，对新法持抵制态度）
- 里正（负责执行，夹在中间）
- 数名贫苦农户（被贷款对象，实际上不理解这笔钱）
- 御史派来的「观察员」（监察新法执行情况）

**戏剧价值**

这是政策「落地」场景，展示了宏观改革在微观层面的变形与扭曲。保甲长的阻挠有其现实利益逻辑；农户的被动有其历史处境；地方官夹在政策压力和实际操作困难之间的处境，是熙宁变法失败根源的缩影。

### 剧本 5：「玄武门前的两个小时」

**场景**：武德九年六月初四清晨，玄武门外，李世民的部队已就位，但门尚未开，李建成和李元吉尚未到来。

**戏剧核心**

等待的两个小时。参与政变的每个人都知道，一旦行动就无法回头；但此时仍可选择撤退。

**人物表**

- 李世民（主视角可选，内心独白为主）
- 尉迟恭（行动派，催促动手）
- 长孙无忌（谋划者，此刻保持沉默）
- 某个动摇的秦王府军士（代表行动者群体的恐惧声音）

**推演问题**

如果主角是这个动摇的军士，他是否有能力和机会阻止或泄露政变？若泄露，历史走向如何改变？（架空模式）

### 剧本 6：「戊戌六君子最后的廷议」

**场景**：1898 年秋，政变后，刑部大牢或审讯场所，谭嗣同等六人被审讯前的短暂聚集。

**戏剧核心**

历史记载谭嗣同有机会出逃却选择留下，其理由是「各国变法，流血者有之，中国未有；此非中国所以不昌也。有之，请自嗣同始」。

这一场景是高度悲剧性的，推演价值在于：让用户体验「明知结局的选择」所包含的历史重量。

---

### 剧本 7：「甲午前夕：北洋舰队的会议室」

**场景**：1894 年，北洋舰队某次军事会议，丁汝昌与几位主要舰长讨论应对日本挑战的战略方案。

**人物表**

- 丁汝昌（北洋水师提督，主视角可选）
- 刘步蟾（主力舰舰长，技术派，强调装备差距）
- 某主战派幕僚（政治压力的代言人）
- 李鸿章的信使（带来含糊指令）

**核心困境**

丁汝昌掌握真实的军事情报（日本海军的实际技术优势），但政治压力要求他保持乐观姿态。他是否如实向上汇报，决定了他对历史的影响。

---

### 剧本 8：「庆历四年秋：范仲淹的最后奏疏」

**场景**：庆历新政走向失败的关键节点，范仲淹决定上疏力争还是主动请辞。

**戏剧核心**

范仲淹的「岳阳楼记」写于被贬之时，「先天下之忧而忧，后天下之乐而乐」是他的政治人格宣言。推演让用户在这一宣言产生的历史节点，体验「坚守vs退让」的真实代价。

---

### 剧本 9：「辛酉政变：恭亲王的选择」

**场景**：咸丰帝刚刚驾崩，热河。慈禧（懿贵妃）秘密派人联络奕訢，提出合作政变。奕訢必须决定是否参与。

**冲突结构**

奕訢与肃顺之间有深刻的个人积怨（政治竞争和满汉路线之争）。但参与宫廷政变，风险极高。慈禧提出的利益交换——以「垂帘听政」换取「议政王大臣」地位——是否值得信任？

**推演问题**

这是典型的「不可信承诺」困境：慈禧和奕訢互相需要对方，但双方都知道合作目标达成后可能翻脸。历史上奕訢最终被慈禧剥夺议政王身份。推演可以模拟奕訢是否可能设计出更有保障的合作条款。

---

### 剧本 10：「绍兴十一年：岳飞与秦桧的最后对话」（架空）

**设计说明**

这是一个架空场景，历史上岳飞被以「莫须有」罪名处死，两人的「最后对话」不见于正史。CourtSim 架空模式可以模拟：若岳飞在最后时刻有机会面见秦桧，他会说什么？秦桧会如何回应？高宗的「12 道金牌」背后的政治逻辑如何在对话中展开？

**价值说明**

架空历史对话的风险在于：可能将历史人物简单化为「善恶两极」。本场景的设计须强调，秦桧的选择有其结构性逻辑（议和是高宗本人的意志），不能简化为个人道德失败。

---

### 剧本 11：「张居正的书房：写给万历的奏疏」

**场景**：张居正改革最艰难的某个夜晚，他独自在书房修改一份可能触怒皇帝的奏疏。

**独特设计**

这是一个「独角戏」场景：主角就是张居正，无对抗人物。戏剧张力来自内心独白——他知道万历正在成长，权力的平衡正在发生微妙变化，他自己的改革与他对皇帝的「管控」之间的矛盾正在激化。

**推演价值**

这一场景模拟的是「成功改革家如何为自己的衰落埋下种子」，是关于权力、傲慢与制度逻辑的深刻命题。

## 七、评测方案

### 7.1 自动化检测（每次 CI 运行）

**JSON 完整性检查**

对所有推演输出的 `dialogueMeta` 字段进行 schema 验证：

```javascript
// 必须存在的字段
const required = ['speakerId', 'stance', 'dialogueType', 'target'];
// dialogueType 枚举值验证
const validTypes = ['支持', '反对', '质疑', '攻讦', '中立陈述', '皇权宣示'];
// 检查 stance 字段格式
assert(typeof meta.stance === 'string' && meta.stance.length > 0);
```

**禁词检测**

建立章回体禁词表，包含：

- 现代情绪词汇：「崩溃」「破防」「内耗」「emo」「躺平」「卷」
- 现代网络词汇：「666」「kk」「没有对比就没有伤害」
- 现代政治话语词汇（应避免出现在古代语境中的现代政治词汇）

**长度合规性检查**

- 单轮角色发言：200–600 字（超出范围触发警告）
- 章回标题：10–20 字
- 章回正文：1500–4000 字（按场景类型可配置）

**年号/人名一致性检查**

维护一份「场景合法人物—年号」对照表，对生成文本中出现的年号和人名进行交叉验证。

### 7.2 人工评测 Rubric

**维度一：历史准确性（1–5 分）**


| 分数  | 标准                            |
| --- | ----------------------------- |
| 5   | 无可识别史实错误，官职、年号、事件完全符合史料       |
| 4   | 有 1–2 处小错误（如官职名称略有出入），不影响核心叙事 |
| 3   | 有明显错误但不影响整体逻辑（如人物年龄偏差较大）      |
| 2   | 有严重错误，可能误导用户对历史的理解            |
| 1   | 存在根本性史实混乱，多朝代概念混用             |


**维度二：戏剧张力（1–5 分）**


| 分数  | 标准                            |
| --- | ----------------------------- |
| 5   | 论辩针锋相对，人物立场清晰，冲突持续升级，皇权介入时机恰当 |
| 4   | 整体有张力，偶有平淡段落                  |
| 3   | 基本完成冲突结构，但论辩套路化明显             |
| 2   | 大部分角色发言趋于雷同，缺乏个性              |
| 1   | 缺乏冲突，像报告文体                    |


**维度三：章回文风（1–5 分）**


| 分数  | 标准                            |
| --- | ----------------------------- |
| 5   | 叙事语言自然得体，古白话与白描手法运用恰当，无现代网文痕迹 |
| 4   | 整体文风合适，有少量现代词汇渗入              |
| 3   | 文言化过度或现代化过度，但不严重影响阅读          |
| 2   | 明显的现代网文化或生硬文言堆砌               |
| 1   | 语言风格与场景完全不符                   |


**维度四：主视角沉浸感（1–5 分）**


| 分数  | 标准                         |
| --- | -------------------------- |
| 5   | 主角信息获取符合历史身份，决策空间真实，后果逻辑自洽 |
| 4   | 整体沉浸，偶有「上帝视角」泄漏            |
| 3   | 信息层级问题明显，主角知道过多本不该知道的信息    |
| 2   | 主视角与叙事框架脱节                 |
| 1   | 主视角机制基本失效                  |


### 7.3 测试用例库（示例）

**用例 TC-001：大礼议年号一致性**

- 输入：嘉靖元年大礼议场景，请求生成杨廷和的第三轮发言
- 期望：发言中出现的年号为「嘉靖」（不得出现「正德」「弘治」等不相关年号）
- 期望：杨廷和使用「皇上」而非「圣上」（明代称谓习惯）
- 验证方式：正则匹配 + 人工复核

**用例 TC-002：皇权介入触发**

- 输入：连续 5 轮未触发皇权介入的推演上下文
- 期望：第 6 轮触发「皇权介入点」提示
- 验证方式：系统日志检查 `emperorInterventionTriggered = true`

**用例 TC-003：章回体禁词**

- 输入：一段包含「破防」「内耗」「卷」的生成文本
- 期望：禁词检测 API 返回 `{ violations: ['破防', '内耗', '卷'] }`
- 验证方式：单元测试

**用例 TC-004：主视角信息隔离**

- 输入：玩家扮演一名普通御史，另一角色进行密室谋划
- 期望：密室谋划内容不出现在主视角的下一轮提示上下文中
- 验证方式：上下文包 diff 检查

## 八、术语表


| 术语                | 解释                                                      |
| ----------------- | ------------------------------------------------------- |
| **廷议**            | 明清两代重大政事由皇帝召集百官公开讨论的制度性会议                               |
| **票拟**            | 内阁对奏疏拟定处理意见（用墨笔写于纸条贴附），皇帝朱批后执行                          |
| **朱批**            | 皇帝用朱砂笔对票拟的最终裁定意见                                        |
| **奏疏**            | 臣下向皇帝上呈的正式文件，按紧急程度有题本、奏本、揭帖等不同形式                        |
| **廷杖**            | 由皇帝命令在朝堂对官员施行杖刑的惩罚制度，明代尤为常见                             |
| **留中**            | 皇帝将奏疏留在内廷不作批示，是一种政治态度的表达                                |
| **台谏**            | 御史台（负责纠察百官）和谏院（负责规谏皇帝）的合称                               |
| **祖宗成法**          | 对开国皇帝及历代先君所定制度的援引，是政治辩论中的最高合法性依据之一                      |
| **年号**            | 皇帝纪年所用的名号，明清一般一帝一年号（除英宗两个年号），故也常以年号称呼皇帝                 |
| **dialogueMeta**  | CourtSim 内部用于描述角色发言元数据的 JSON 字段，包含发言者、立场、对话类型等          |
| **scenario**      | CourtSim 中一个完整的推演场景配置，包含朝代、时间范围、人物表、初始事件等               |
| **simulationRun** | 对一个 scenario 的一次完整推演实例                                  |
| **主视角角色**         | 用户在推演中代入体验的历史人物，通过 `scenario.protagonistCharacterId` 指定 |
| **皇权介入点**         | 推演引擎在特定条件下触发的皇帝行动时机，用于防止推演陷入无实质进展的循环                    |
| **夺门之变**          | 1457 年英宗复辟事件，见本文 3.1.1                                  |
| **大礼议**           | 1521–1536 年嘉靖朝礼制争议，见本文 3.3.1                            |
| **严格模式**          | CourtSim 三种仿真模式之一，要求所有内容有史料依据                           |
| **演绎模式**          | CourtSim 默认模式，允许在史实框架内合理推演                              |
| **架空模式**          | 允许改变历史结果的假设推演模式                                         |
